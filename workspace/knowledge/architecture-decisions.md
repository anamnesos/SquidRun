## Session 399 — Changed Behavior (native DB ABI drift)

Durable behavior changes from commit `104fbd64`. Restart required to activate — the live session-399 main predates the binary fix and holds the broken in-memory state.

1. **Native-module ABI drift root cause.** The Electron main process runs Electron 28.3.3 / Node **18.18.2 / ABI 119**, which lacks `node:sqlite`; CLI scripts run system Node **24.x / ABI 137** with `node:sqlite` built in. All three main-process stores (evidence-ledger `evidence-ledger-store.js`, supervisor `supervisor/store.js`, cognitive-memory via `sqlite-compat.js`) share a `node:sqlite`-first → `better-sqlite3`-fallback `loadSqliteDriver`, so inside Electron they ALL depend on `better-sqlite3`. A stray `npm install`/`rebuild` under Node 24 recompiles `better_sqlite3.node` to ABI 137, which Electron 28 (ABI 119) then **cannot load** → every store silently enters degraded mode (reads AND writes become no-ops; there is NO write-outbox, so degraded writes are dropped, not queued). CLI scripts keep working (node:sqlite), so the on-disk journal stays populated while the running app is blind to it.

2. **Symptom decoupling — the reply-guard false-nag was downstream, not a guard bug.** A delivered `hm-send telegram` row written by the CLI (Node 24) is fully matchable (`channel=telegram`, `direction=outbound`, `status=acked`, `ack_status=telegram_delivered`, `target=user`, metadata `chatId`), but the main-process reply-guard reconciler could not query the journal at all, so it kept nagging. **Triage rule:** matchable journal row + persistent nag ⇒ suspect main-process SQLite / native-ABI health (probe `better-sqlite3` under Electron) BEFORE shipping a duplicate Telegram reply or "fixing" reply-guard logic. (Triage also in `workflows.md`.)

3. **Postinstall self-probing rebuild guard (`ui/scripts/postinstall-electron-rebuild.js`).** Wired as `ui/package.json` `postinstall`, so it runs on `npm install`/`npm ci` ONLY — never on app launch/relaunch. It first probes `better-sqlite3` in a child Electron run (`ELECTRON_RUN_AS_NODE=1`); if it loads at ABI 119 it skips, and only on probe failure does it run the targeted `electron-rebuild -f -o better-sqlite3` and re-probe. This is the **durable protection**: the rebuilt binary lives in local `node_modules` (NOT git-tracked), so the guard — not the commit — is what re-heals ABI drift on every machine and after every future Node-24 install.

4. **Restart-safety profile (verified S399 Oracle gate).** Clean quit→relaunch is lock-safe: relaunch never rebuilds, so it never unlinks the loaded `.node` (the EPERM Builder saw only occurs when a rebuild tries to replace the binary while a process holds it). ## Session 423 — Changed Behavior (restart-continuity resume seam)

Durable decisions from commit `068ba2e2` (Oracle restart-risk gate: GO). Module mechanics are in `ARCHITECTURE.md` (`cli-resume-invocation.js`, `pane-session-id-store.js`); this records the rationale + the operational restart-timing rule.

1. **Root cause of "arms respawn cold."** An external SquidRun restart (`hm-restart-execute.js`) is a brute-force process-TREE kill: `shutdownElectronProcesses` SIGTERMs every Electron descendant, which includes the detached terminal-daemon + its node-pty children + the CLI subprocess. On Windows `process.kill(pid,'SIGTERM')` = uncatchable `TerminateProcess`, so the daemon's own SIGTERM save handler does NOT run during a restart — only the 30s periodic scrollback save (`terminal-daemon.js`) survives, and that is TEXT only. In-flight CLI conversation state is unrecoverable from SquidRun's side. (Note the latent contradiction: the daemon is spawned `detached` "to survive parent exit" and `daemon-client.connect()` prefers an existing daemon, but the tree-kill + generation-reap deliberately defeat survival because a surviving daemon = stale code behind the pidfile.)

2. **Fix = id-addressed CLI resume, NEVER cwd-scoped.** Resume the CLI's OWN prior session by a per-pane id. cwd-scoped resume (`claude --continue`, `codex resume --last`) is FORBIDDEN because the 3 core panes all run claude in `D:\projects\squidrun` (paneProjects null) and the 4 TrustQuote arms all run codex in `D:\projects\TrustQuote` — cwd-scoped "resume most recent" cross-wires every pane in a shared dir onto one session. claude: `--session-id <uuid>` creates, `-r/--resume <uuid>` resumes, gated on session EXISTENCE in `~/.claude/projects/<encodedCwd>/<uuid>.jsonl` (NOT the scrollback snapshot — different signal). `--fork-session` is forbidden (mints a new id, breaks stability). codex arms COLD by design: codex has resume-by-id but NO pin-at-creation flag, so a shared-cwd arm's id can't be deterministically set; "adopt most-recent existing session" is rejected because it reintroduces the collision. Revisit arms only if codex adds pin/name-at-creation or content-based disambiguation.

3. **Operational rule — one-restart lag (do NOT burn a dedicated restart for resume).** The fix is restart-READY but it does NOT preserve the CURRENT session across the NEXT restart: the conversations running now were created with claude's own auto-generated ids, not the per-pane UUIDs this seam mints, so the activating restart cold-starts them (`--session-id <freshUuid>` = new conversation). Resume only pays off from restart #2 onward, once stored ids match real sessions. Therefore: let the resume fix ride the next restart that happens for some OTHER reason — never trigger a restart solely to activate it (zero benefit to the live session, which it cold-starts). This is the unavoidable cost of id-pinning, not a defect.

4. **Gate proof method (reusable).** Resume-by-id across a process death was proven without an app restart: `claude --session-id <uuid> --print` in one process, then `claude --resume <uuid> --print` in a SEPARATE process, recalled a planted codeword. The separate-process boundary IS the restart case. The only unproven sliver — interactive TUI visually repainting old message bubbles — is cosmetic and already double-covered by the scrollback-hydration path, so it does not gate restart-readiness.
