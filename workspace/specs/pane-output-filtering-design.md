# Pane Output Filtering Design

## Origin

Session 258. Architect approved a two-lane approach for hiding internal agent output from user-visible pane rendering without relying on heuristic text guessing.

## Problem

Agent panes currently receive PTY output as raw text chunks. The renderer can decorate lines, but it does not yet know whether a given chunk is user-facing content or internal coordination/self-talk. That makes it easy for internal notes, debug lines, or routing hints to appear in the visible pane.

## Current Constraints

- `pty-data-*` events are plain text today, not structured visibility-aware envelopes.
- `inject-message` already carries structured payload metadata and is the best path for app-originated messages we control.
- CSS-only filtering is not sufficient. The terminal path is text/canvas oriented, and decoration is not the same as semantic suppression.
- Heuristic filtering is explicitly rejected. We should not guess based on wording like "thinking", "maybe", or "debug".

## Approved Approach

### Lane 1: Structured visibility for injected app messages

For messages sent through the existing `inject-message` path, add a visibility field in message metadata:

```json
{
  "meta": {
    "visibility": "user"
  }
}
```

Supported values:

- `user`: render normally
- `internal`: do not render in the user pane

This lane covers messages generated by the app/runtime where we already control the payload format.

### Lane 2: Explicit sentinel markers for PTY output

For agent-authored PTY output, use an explicit sentinel prefix for internal-only lines, for example:

```text
[[SR_INTERNAL]] coordinating with Architect about queue state
```

The renderer should strip or suppress lines carrying this sentinel before they are written to the terminal.

This lane covers raw PTY text where no structured visibility metadata exists yet.

## Renderer Integration

Implement one shared output-filtering helper and use it in both pane renderers:

- `ui/modules/terminal.js`
- `ui/pane-host-renderer.js`

The shared helper should:

1. Hide `inject-message` payloads whose `meta.visibility` is `internal`
2. Suppress PTY lines marked with the internal sentinel
3. Leave all other output unchanged

Using one helper avoids drift between the main pane renderer and the hidden pane host.

## Non-Goals

- No heuristic text classification
- No CSS-only hiding strategy
- No attempt in this pass to redesign PTY transport around fully structured event envelopes

## Follow-On Work

If we later want richer semantics, the long-term upgrade path is structured PTY/event envelopes with explicit message types and visibility. The two-lane design above is the minimal reliable step that fits the current architecture.
