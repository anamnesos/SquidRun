# Mira Import Approval Marker Semantics v0

Status: approval marker design only. No marker instance, apply code, copy,
import, queue mutation, or runtime continuity loading is introduced here.

A future first import lane must require an approval marker before copying any
file from a reviewed report into `MIRA_STATE_ROOT`.

## Marker Location

Approval markers should be stored beside import reports:

```text
mira/imports/approvals/<approval_id>.json
```

The marker is a source-control reviewed artifact. It is not generated by the
dry-run executor and it is not inferred from memory, recall, chat, runtime
health, or report existence.

## Required Meaning

An approval marker may approve only:

- the named report id and report path;
- the named batch id;
- the exact listed record ids;
- copy-only execution.

It must not approve:

- moving files;
- deleting files;
- mutating the import queue;
- mutating the report;
- loading imported continuity into runtime.

## First Batch Candidate

The current expected first marker, if later approved, would target:

- report id `first-batch-dry-run-v1`;
- batch id `acceptance-permission-contracts-v1`;
- records `presence_runtime_acceptance`, `north_star_acceptance`, and
  `pc_embodiment_permission`.

This document does not create that marker. It only defines the shape and limits
the marker must obey.

## Non-Scope

- No approval marker instance.
- No apply/import execution.
- No copied files.
- No queue status mutation.
- No runtime auto-load of imported continuity.
- No bridge, UI, or Telegram route work.
